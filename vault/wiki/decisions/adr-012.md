# ADR-012: Extension-Based Harness Integration via Event Bus Layering

## Context

The HARNESS-PRD specifies an 8-layer mandatory execution pipeline. The pi coding agent provides an `ExtensionAPI` with 5 native events: `session_start`, `session_compact`, `session_shutdown`, `before_agent_start`, and `tool_result`. The PRD requires 30+ lifecycle hooks (P33: `before_llm_call`, `after_tool_call`, `before_edit`, `after_edit`, phase-transition gates, etc.).

Three integration paths were considered:
- **A) Fork pi to add hook points.** Maintenance burden. Blocks on upstream.
- **B) Wrap pi at process boundary.** Fragile, breaks on pi updates.
- **C) Build a harness event bus extension that layers richer hooks on top of pi's 5 native events.** Zero pi changes. Full control within extensions.

## Decision

**Use an extension-based harness event bus. No fork. No process wrapping.**

A single `harness-event-bus` extension subscribes to pi's 5 native events and fires a richer set of harness-level hooks internally. State machine tracks pipeline position (L1→L2→L3→L4→L5-L8). Phase transitions detected via tool result patterns. `before_agent_start` becomes the injection point — harness rewrites system prompt based on current pipeline state.

### Event Mapping

| Pi Native Event | Harness Hooks Fired |
|----------------|---------------------|
| `before_agent_start` | `before_llm_call`, `spec_gate_check`, `plan_gate_check`, `drift_correction_inject`, `l4_gate_block`, `l4_critic_inject`, `anxiety_guard_check` |
| `tool_result` | `after_tool_call`, `after_edit`, `after_read`, `after_bash` (filtered by `event.toolName`), `phase_transition_detect`, `drift_pattern_detect` |
| `session_start` | `session_init`, `wiki_bootstrap`, `spec_load`, `hot_cache_inject` |
| `session_compact` | `context_compacted`, `harness_state_reinject` |
| `session_shutdown` | `session_teardown`, `consensus_flush`, `keep_rate_sample` |

### Enforcement Model

Cannot literally block LLM calls (pi runtime controls invocation). Enforcement is prompt-level:
- L1 gate: system prompt = clarification loop until spec hardened. ~95% compliance via directive prompting.
- L2.5 drift: `tool_result` pattern analysis → drift flag → next `before_agent_start` injects correction nudge.
- L4 gate: if critic finds failure, system prompt becomes "fix these specific failures" instead of open-ended agent prompt.
- Phase 16 gate: deterministic tools (`biome check`, `tsc --noEmit`) run in `tool_result`. If fail, prompt says "lint/type errors found. Fix before proceeding."

Not 100% software-enforced, but directive prompting achieves high compliance. L7 orchestration (later phase) may add process-level enforcement via `pi.exec()`-based gate scripts.

## Rationale

- **Zero pi dependency**: Works with pi v0.70.2 today. No upstream PRs, no fork maintenance.
- **All harness logic in one extension**: The event bus is a single `.pi/extensions/harness-event-bus.ts`. Internal hooks are just function calls within the extension module.
- **Proven pattern**: `wiki-hooks.ts` already demonstrates injecting system prompts via `before_agent_start` and acting on `tool_result`. The harness event bus is the same pattern, scaled.
- **Upgrade path**: If pi adds native hooks later, the event bus can switch from pattern-detection to native events without changing harness logic.

## Consequences

### Positive
- Ships immediately. No external dependencies beyond pi.
- Harness is self-contained in `.pi/extensions/`.
- Event bus can add hooks without pi changes.

### Negative
- ~95% compliance, not 100%. LLM can ignore directive prompts.
- Pattern detection in `tool_result` is heuristic, not guaranteed.
- No true `before_llm_call` interception — can only inject into system prompt, not prevent the call.

### Mitigations
- L7 orchestration (P23) adds process-level enforcement via `pi.exec()` gate scripts.
- Multiple defense layers (L1 hardening + L2.5 drift + L4 adversarial + P16 deterministic) mean a single-layer bypass is caught downstream.
- Compliance monitoring in L5 observability tracks bypass rates per layer.
